## 들어가며

**3년치 거래처 계약서 50개 중에서 위약금 조항을 찾는 데 얼마나 걸립니까.**

파일 하나씩 열어보거나, 검색어로 찾아보거나, 담당자에게 물어보거나. 어떤 방법이든 빠르지 않습니다. 결국 찾다가 포기하고 "그냥 새로 씁시다"가 되거나, 한 시간 뒤에 겨우 찾아내거나.

컨설팅사 North Highland는 Claude Projects 도입 후 이런 작업 속도가 **5배 빨라졌다**고 밝혔습니다.

방법은 단순합니다. 찾아야 할 문서를 미리 Claude에 올려두는 것입니다.

이 글에서는 Claude Projects가 뭔지, 어떻게 설정하는지, 법무·기획·영업 직군이 실제로 어떻게 쓰는지를 직접 써본 경험을 바탕으로 정리합니다.

## Claude Projects, 이게 뭔가요

> **[이미지 지시]** Claude.ai Projects 탭 첫 화면 — claude.ai 접속 → 왼쪽 사이드바에서 'Projects' 탭 선택. 프로젝트 목록이 보이는 화면 전체 캡처. 프로젝트가 몇 개 있는 상태면 더 좋음.

한 줄로 설명하면 이렇습니다. "내가 자주 쓰는 문서와 정보를 Claude에 저장해두는 공간"입니다.

일반 Claude 채팅은 대화가 끝나면 아무것도 남지 않습니다. 오늘 계약서를 분석했어도, 내일 다시 접속해서 새 채팅을 열면 Claude는 기억하지 못합니다.

Projects는 다릅니다. 올려둔 문서가 사라지지 않습니다. 내일 접속해도, 한 달 뒤에 접속해도 같은 문서가 그대로 있습니다. Claude는 그 문서를 읽은 상태로 대화를 시작합니다.

계약서를 한 번 올려두면, 다음부터는 "A사 계약서에서 위약금 조항 찾아줘"라고 바로 물어볼 수 있습니다. 쌓이면 찾기 어려운 정보들을 올려두고, 한번에 전부 찾아내게 할 수 있습니다. 이 project만을 위한 지침도 따로 만들어줄 수 있죠.

## Projects에는 생각보다 많은 정보를 저장할 수 있습니다

> **[이미지 지시]** 200K 컨텍스트 시각화 — Canva에서 직접 제작. "A4 500페이지 = 200K 토큰" 비교 인포그래픽. 또는 Claude Projects에 파일 여러 개 올라간 상태 캡처.

Projects에는 한 번에 얼마나 올릴 수 있을까요. **200K 토큰**입니다. 숫자만 보면 감이 안 옵니다. 구체적으로 환산하면 이렇습니다.

- A4 용지 기준 **약 500페이지 분량**
- 일반 계약서 10페이지짜리 **50개**
- 주간 회의록 2페이지짜리 **250주치 (약 5년)**
- 표준 제안서 20페이지짜리 **25개**

이 분량을 한 번에 올려두고 Claude에게 질문할 수 있습니다. Claude는 그 문서 전체를 읽은 상태에서 답합니다.

## Projects 설정하는 법 (처음 해보는 분용)

> **[이미지 지시]** 새 프로젝트 만들기 화면 — Claude.ai > Projects 탭 > '+ 새 프로젝트' 버튼 클릭 후 나오는 이름 입력 화면. 입력란이 비어있는 상태로 캡처.

처음이라도 10분이면 됩니다.

### 1단계: Projects 탭 열기

Claude.ai 접속 → 왼쪽 사이드바에서 **Projects 탭** 선택 → '+ 새 프로젝트' 클릭

### 2단계: 프로젝트 이름 짓기

목적에 맞게 구체적으로 짓는 게 좋습니다.

- "계약서 모음" 말고 → "거래처 계약서 2024-2025"
- "회의록" 말고 → "A프로젝트 회의록"
- "고객사" 말고 → "B사 영업 히스토리"

프로젝트가 5개 이상 생기면 이름이 중요해집니다.

### 3단계: 문서 올리기

> **[이미지 지시]** 파일 업로드 화면 — 프로젝트 안에서 '파일 추가' 또는 클립 아이콘 클릭 후 파일 선택 화면. PDF나 DOCX 파일 아이콘이 보이는 상태로 캡처.

프로젝트 안에서 파일 첨부 버튼 클릭 → 문서 선택 → 업로드

지원 형식은 **PDF, DOCX, TXT** 등 주요 문서 파일 전부 됩니다. 파일당 30MB까지 올릴 수 있습니다.

텍스트를 직접 붙여넣는 것도 됩니다. 사내 규정, 팀 가이드라인, 자주 쓰는 정보를 텍스트로 저장해두면 됩니다.

### 4단계: 바로 질문하기

> **[이미지 지시]** 프로젝트 안에서 대화 화면 — 파일이 올라간 프로젝트에서 질문하고 답변 나온 화면. 파일 아이콘과 대화창이 함께 보이면 좋음. 직접 캡처.

파일 올라간 상태에서 채팅창에 바로 질문하면 됩니다. Claude가 올려둔 문서 전체를 읽고 답합니다.

## 직무별 실전 활용법

### 방법 1) 법무·계약 담당자: 계약서 더미에서 해방되는 법

> **[이미지 지시]** 계약서 파일 올리고 질문하는 화면 — Projects 탭에서 PDF 파일 여러 개 올라간 상태에서 조항 관련 질문 입력 중인 화면. 직접 캡처.

계약서 관련 업무의 시간 대부분은 찾는 것에 씁니다. 특정 조항이 어디 있는지, 이번 계약서와 지난번 계약서가 어떻게 다른지, 이 조건이 다른 거래처에도 적용되는지. (그리고 찾다가 결국 법무팀에 전화하죠 ...)

**기존 방식**

계약서 폴더 열기 → 파일 하나씩 클릭 → Ctrl+F로 키워드 검색 → 없으면 다음 파일 → 반복 → 포기 또는 발견 (평균 30분~1시간)

**Projects 사용 후**

계약서 전체 올려두기(1회) → 이후엔 질문만 (1~2분)

> "을의 손해배상 책임이 명시된 계약서 모두 찾아줘"
> "A사와 B사 계약서에서 납기 조항 차이를 비교해줘"
> "2024년 이후 체결된 계약서 중 자동 갱신 조항이 없는 것 알려줘"
> "이 조항이 표준 계약서 대비 불리한 조건인지 검토해줘"

올려두기 좋은 문서들:
- 거래처별 계약서 전체
- 표준 계약서 양식 (비교 기준)
- 법무 검토 의견서
- 클레임·분쟁 이력 메모

### 방법 2) 기획자: 회의록이 쌓일수록 강해지는 맥락

> **[이미지 지시]** 회의록 여러 개 올린 Projects 화면 — 파일 목록에 날짜별 회의록 파일 이름이 여러 개 보이는 상태. 직접 캡처.

기획 업무에서 반복되는 비효율이 있습니다. 지난 회의에서 뭘 결정했는지 다시 확인하는 것입니다. 회의록을 뒤지거나, 담당자에게 물어보거나, 기억에 의존합니다.

**기존 방식**

"3기 때 이 모듈 빼기로 했던 이유가 뭐였지?" → 회의록 폴더 탐색 → 날짜 추정 → 파일 열기 → 검색 → 못 찾으면 Slack 검색 → 없으면 포기

**Projects 사용 후**

회의록 쌓아두기(누적) → 그냥 물어보기

> "지난 분기에 A 기능 개발 보류 결정한 이유가 뭐였지?"
> "이번 프로젝트에서 고객사가 요청한 수정 사항 전부 뽑아줘"
> "지난 3개월간 회의에서 반복적으로 나온 이슈가 뭔지 요약해줘"
> "이번 기수 커리큘럼 변경 내역을 시간순으로 정리해줘"

회의록이 쌓일수록 더 정교한 답이 나옵니다. 맥락이 누적되기 때문입니다. 프로젝트 기간이 길어질수록 이 도구의 가치가 올라갑니다.

올려두기 좋은 문서들:
- 주간/월간 회의록 전체
- 의사결정 로그 (무엇을, 왜 결정했는지)
- 기획안 버전별 이력
- 고객사·이해관계자 피드백 메모

### 방법 3) 영업 담당자: 고객사별 히스토리를 기억하는 AI

> **[이미지 지시]** 고객사별 Projects 목록 화면 — Projects 탭에서 고객사 이름으로 된 프로젝트 여러 개가 목록에 보이는 화면. 직접 캡처.

영업 담당자가 고객사를 인수인계받을 때 가장 힘든 것은 히스토리 파악입니다. 이전 제안서, 미팅 메모, 클레임 내용, 특이 요청사항이 여기저기 흩어져 있습니다. 전임자한테 일일이 물어보거나, 없으면 처음부터 파악해야 합니다.

**기존 방식**

인수인계 문서 읽기 → 이메일 쓰레드 뒤지기 → 전임자에게 전화 → 미팅 나가서 어색하게 파악

**Projects 사용 후**

고객사별 프로젝트 만들기 → 관련 문서 모두 올리기 → 미팅 전에 질문해서 파악

> "지난 2년간 이 고객사가 거절한 이유를 정리해줘"
> "기존 제안서 기반으로 이번 미팅 브리핑 자료 만들어줘"
> "이 고객사가 민감하게 반응한 조건이 뭐였지?"
> "이번 분기 이 고객사에 제안할 수 있는 내용 추려줘"

새로 합류한 담당자도 히스토리를 처음부터 파악할 필요가 없습니다. **고객사 문서가 곧 인수인계 자료가 됩니다.**

올려두기 좋은 문서들:
- 미팅 메모 전체
- 제안서 버전별 이력
- 클레임·특이 요청 기록
- 고객사 내부 조직도 또는 담당자 메모

## 직접 써봤습니다

> **[이미지 지시]** 실제 사용 중인 Projects 화면 — Projects 탭에서 본인 프로젝트 목록 또는 특정 프로젝트 안의 파일 목록 캡처. 직접 쓰는 화면이면 됨.

6개월짜리 프로젝트 중간에 팀이 바뀌었습니다.

새로 합류한 팀원이 "이 방향 왜 이렇게 잡은 거예요?"라고 물었을 때, 저도 정확히 기억이 안 났습니다. 회의록 폴더 뒤지고, Slack 검색하고, 결국 당시 담당자에게 따로 물어봤습니다.

그때부터 회의록이랑 기획 문서를 Projects에 올려두기 시작했습니다.

가장 달라진 건 **"왜 이렇게 됐지?" 질문에 바로 답할 수 있게 된 것**입니다. "지난달에 A안 말고 B안으로 가기로 한 이유가 뭐였지?" — 이제 그냥 물어보면 됩니다. 회의록 뒤질 필요가 없습니다.

두 번째 변화는 **중간 보고 준비 시간**입니다. 이전엔 지난 회의 내용 정리하고 진행 경과 묶는 데 반나절 걸렸습니다. 지금은 "지난 두 달 진행 경과 요약해줘"라고 하면 초안이 나옵니다.

Projects 쓰면서 깨달은 건, 문서를 열심히 만들어놓고 정작 못 찾아서 못 쓰는 게 진짜 문제였다는 겁니다. (만들어놓고 묻어두는 거 저만 그런 거 아니죠 ...)

## 알아두면 좋은 것

> **[이미지 지시]** Skills와 Projects 탭 비교 화면 — Claude.ai 사이드바에서 Projects 탭과 Skills 탭이 모두 보이는 화면 캡처. 또는 각각 탭 클릭한 화면 두 장을 나란히.

**올리는 문서의 질이 결과를 결정합니다.** 정리되지 않은 메모나 불완전한 문서가 많으면 답도 불명확하게 나옵니다. 올리기 전에 한 번 정리하는 것이 좋습니다.

**PDF 100페이지 이하일 때 가장 정확합니다.** 공식 스펙 기준으로 100페이지 이하 PDF는 텍스트와 이미지를 모두 분석합니다. 1000페이지가 넘어가면 텍스트만 처리합니다. 두꺼운 문서는 챕터별로 나눠서 올리는 것이 낫습니다.

**문서가 업데이트되면 직접 교체해야 합니다.** 자동 동기화가 없습니다. 계약서가 수정됐거나 새 회의록이 추가됐을 때 Projects에도 반영해야 합니다.

**Claude Projects와 Skills는 함께 쓸 때 제대로 됩니다.**

| 기능 | 저장하는 것 | 예시 |
|------|------------|------|
| **Projects** | 정보·맥락 | "거래처 계약서, 회의록, 고객사 히스토리" |
| **Skills** | 작업 방식 | "이 문서를 요약할 때 항상 이 형식으로" |

Projects가 "무엇을 아는지"라면, Skills는 "어떻게 할지"입니다. 문서를 담아두는 것은 Projects, 그 문서로 일하는 방식을 정해두는 것은 Skills입니다. (Skills 활용법은 https://aistolenmyjob.tistory.com/1 에서 따로 정리했습니다)

## 마무리

문서가 많은 업무일수록, 히스토리가 중요한 업무일수록 Projects의 가치가 올라갑니다.

계약서 50개를 올려두고 조항을 물어보는 것, 3년치 회의록을 넣어두고 맥락을 꺼내는 것, 고객사 히스토리를 모아두고 브리핑 자료를 뽑는 것. 문서를 들고 다니는 방식이 달라집니다.
